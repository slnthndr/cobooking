# 🏢 CoBooking — Enterprise Microservices Platform

**CoBooking** — это учебная backend-платформа для онлайн-бронирования рабочих мест (коворкингов). 
Проект создан для глубокого погружения в микросервисную архитектуру и демонстрирует реализацию сложных enterprise-паттернов в реальных условиях.

## 🚀 Чему учит этот проект?
Здесь реализованы подходы, используемые в HighLoad системах:
1. **API Gateway Pattern:** Единая точка входа, маршрутизация, централизованная проверка JWT токенов.
2. **Cache-Aside Pattern:** Использование **Redis** для молниеносной отдачи данных и снижения нагрузки на БД.
3. **ACID Транзакции:** Защита от состояния "гонки" (Race Condition) и двойного бронирования на уровне **PostgreSQL**.
4. **Event-Driven Architecture (EDA):** Асинхронное взаимодействие микросервисов через брокер сообщений **RabbitMQ**.
5. **Object Storage:** Работа с S3-совместимым хранилищем **MinIO** для загрузки пользовательских медиа-файлов.
6. **Observability (Наблюдаемость):** Сбор метрик в **Prometheus**, визуализация в **Grafana** и распределенная трассировка запросов через **Jaeger (OpenTelemetry)**.

## 🏗 Архитектура

Платформа состоит из 1 API Gateway и 5 независимых микросервисов (написаны на Go 1.22):

* `api-gateway` (:8080) — Проксирует запросы, проверяет токены, собирает метрики.
* `ms-auth` — Регистрация, логин, выдача JWT (Bcrypt hashing).
* `ms-places` — Каталог рабочих мест. Кэширует ответы в Redis, грузит фото в S3.
* `ms-booking` — Ядро бизнес-логики. Создает бронирования, блокирует пересечения по времени.
* `ms-payments` — Имитация оплаты, прием вебхуков от банка.
* `ms-notifications` — Фоновый воркер. Слушает RabbitMQ и "отправляет" email-уведомления.

## 🛠 Быстрый запуск

Вам понадобятся только установленные `Docker` и `Docker Compose`.

1. Клонируйте репозиторий:
```bash
git clone https://github.com/ВАШ_НИК/cobooking.git
cd cobooking
```

2. Запустите весь кластер (12 контейнеров) одной командой:
```bash
docker compose up --build -d
```
*(Первый запуск займет пару минут, так как Docker будет компилировать Go-бинарники).*

## 🎓 Учебная методичка: Как тестировать архитектуру?

Для удобного тестирования в корне проекта лежит файл **`cobooking-full-api.yaml`**. 
Откройте **Postman**, нажмите `Import` (Ctrl+O) и выберите этот файл. Слева появится готовая коллекция запросов.

### Сценарий 1: JWT Авторизация
1. Откройте папку `Users` -> `POST Регистрация`. Нажмите **Send**. Пользователь сохранится в PostgreSQL.
2. Откройте `Auth` -> `POST Вход`. Нажмите **Send**.
3. В ответе придет `accessToken`. **Скопируйте его**. Этот токен нужно будет вставлять во вкладку `Authorization` -> `Bearer Token` для всех защищенных запросов ниже.

### Сценарий 2: Тестируем кэш Redis (Cache-Aside)
*Задача: Увидеть, как Redis ускоряет отдачу данных.*
1. Откройте терминал и запустите просмотр логов сервиса мест: `docker logs -f cobooking-ms-places`.
2. В Postman откройте `Places` -> `GET Получение информации о рабочем месте`. (В URL укажите ID = 1).
3. Нажмите **Send**. В терминале вы увидите: `[CACHE MISS] Fetching from DB...` (Сервис сходил в PostgreSQL и сохранил результат в Redis).
4. Нажмите **Send** еще раз. В терминале появится: `[CACHE HIT]`. Запрос вообще не трогал базу данных, а вернулся за микросекунды из оперативной памяти!

### Сценарий 3: Защита от двойного бронирования (ACID)
*Задача: Попытаться сломать систему, забронировав одно место на одно время дважды.*
1. Вставьте ваш `accessToken` в авторизацию.
2. Откройте `Bookings` -> `POST Бронирование рабочего места` (URL: `/api/v1/bookings/1`).
3. Нажмите **Send**. Бронь создана (201 Created).
4. **Не меняя время**, нажмите **Send** еще раз.
5. Вы получите ошибку **409 Conflict** (`SLOT_ALREADY_RESERVED`). Транзакция в PostgreSQL распознала пересечение диапазонов времени и отменила запись!

### Сценарий 4: Асинхронность (RabbitMQ)
*Задача: Увидеть, как сервисы общаются без прямых HTTP-запросов.*
1. Откройте логи сервиса уведомлений: `docker logs -f cobooking-ms-notifications`.
2. В Postman измените время бронирования (чтобы обойти защиту из Сценария 3) и создайте новую бронь.
3. Как только бронь создастся, `ms-booking` моментально кинет JSON в RabbitMQ.
4. Вы увидите в логах терминала красивое сообщение: `📩 SENDING EMAIL TO USER...`. Сервис уведомлений поймал сообщение из очереди и обработал его в фоне.

### Сценарий 5: Object Storage (S3 / MinIO)
1. Вставьте токен в авторизацию.
2. Откройте `Places` -> `POST Загрузка фото` (URL: `/api/v1/places/1/image`).
3. Перейдите во вкладку **Body** -> выберите **form-data**.
4. В ключе `image` выберите тип **File** (справа в скрытом меню) и прикрепите любую картинку с компьютера.
5. Нажмите Send. Вы получите прямую ссылку на картинку в вашем личном S3 облаке.

## 📋 Информация для Системных Аналитиков (SA)

Архитектура проекта спроектирована на основе детального технического задания и покрывает ключевые бизнес-требования платформы «CoBooking».

### 1. Бизнес-цели системы
* Повышение удобства и скорости онлайн-бронирования рабочих мест.
* Увеличение загрузки рабочих пространств (Occupancy rate) и прибыльности.
* Снижение операционных издержек за счет автоматизации (оплаты, уведомления).

### 2. Статусная модель (Жизненный цикл)
* **Бронь (Booking):** `created` ➔ `reserved` ➔ `paid` ➔ `completed` (или `cancelled` / `no_show`).
* **Платеж (Payment):** `pending` ➔ `completed` (или `failed` / `refunded`).

### 3. Бизнес-правила и Ограничения (Business Rules)
* **Double-booking protection:** Система гарантирует невозможность бронирования одного и того же слота (пересечение `bookingStartTime` и `bookingEndTime`) двумя пользователями благодаря ACID-транзакциям на уровне базы данных.
* **Eventual Consistency:** Обновление доступности мест и рассылка уведомлений происходят асинхронно через RabbitMQ, не блокируя пользовательский путь (UX).
* **JWT Lifecycle:** Доступ к системе (Access Token) выдается на 15 минут для повышения безопасности. Refresh Token хранится в Redis и живет 7 дней.
* **Rate Limiting:** Защита от DDoS и Brute Force атак реализована на стороне API Gateway.

### 4. Стратегия мониторинга бизнес-метрик (Продуктовая аналитика)
Архитектура позволяет собирать следующие продуктовые метрики для дашбордов:
* **Конверсия поиска в бронь (Booking conversion rate):** `bookingCreated / search`
* **Конверсия брони в оплату (Payment conversion):** `paymentCompleted / bookingCreated`
* **Загрузка рабочих мест (Occupancy rate):** `total booked time / total available time`
* **Процент неявок (No-show rate):** `noShowDetected / bookingCreated` (используется для автоматического внедрения штрафов или предоплаты).

### 5. Точки интеграции (Внешние системы)
* **Оплата:** Интеграция с платежными шлюзами (Банки / СБП) по паттерну Webhook.
* **Авторизация:** Поддержка OAuth 2.0 / OIDC (Google, VK).
* **Уведомления:** Абстрагированный брокер для подключения SMS, Push-серверов и Email-шлюзов.
